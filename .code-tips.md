# Code Tips (written by reviewer only; workers read-only)

## Invariants
- [2026-06-11, task-001] The worker briefing is EXACTLY `tips + "\n\nTASK: " + brief`. The `"\n\nTASK: "` separator is a fixed literal and is always present, even when `.code-tips.md` is absent (then the briefing begins with `\n\nTASK: `). Do not special-case the no-tips path to a bare `"TASK: "`.
- [2026-06-11, task-001] Feed the task brief contents verbatim into the briefing. Do not `.strip()`/`.rstrip()` the brief or the tips text before composing.
- [2026-06-11, task-001] Code in this repo must never write OR commit `.code-tips.md`; it is reviewer-authored and worker-read-only. Do not scaffold/commit an empty `.code-tips.md` from a worker task either.
- [2026-06-11, task-001, task-002] run.json field names are an EXACT contract — add only the fields the brief names, nothing extra. Top-level `task_id, started_at, finished_at, base_commit, workers[]`; M2 adds top-level `review` ({exit_code, verdict{outcome, candidate, reasoning}} or null) and per-worker `candidate_label` ("A".. or null). Each worker also `id, branch, worktree, exit_code, stdout_file, gate{passed, results[]{name, exit_code, autopsy}}`. `stdout_file` is the bare relative name `"w1.stdout"`; `worktree` is `"../<task-id>-<id>"`. Do NOT add unrequested keys (e.g. a top-level `label_mapping`) — it is redundant with `candidate_label` and breaks the contract. On-disk run.json must equal the returned dict, so strip any internal-only keys (e.g. an `_worktree_abs`) before serializing.
- [2026-06-11, task-001] All gate commands run in order even if an earlier one fails (no short-circuit); `gate.passed` is the AND of all gate exit codes being 0.
- [2026-06-11, task-002] M2 anonymization: candidates = gate-passing workers; assign labels A,B,C in RANDOM order (`random.shuffle`); write `git diff <base>..<HEAD>` to `candidates/<LABEL>.diff`. The label->worker mapping must NEVER reach the reviewer: not in the briefing string, not in diff filenames/contents, not in any file the reviewer (CWD=repo toplevel) can read. Record it only AFTER the reviewer exits, and only as the per-worker `candidate_label` in run.json. A separate `candidate-mapping.json` audit file is tolerated only if written post-reviewer, but it is not required and collides in name with task-001 attribution artifacts — prefer not to.
- [2026-06-11, task-002] Review briefing (substituted into the reviewer's `{prompt}`), in order: task brief verbatim; per candidate the LABEL + relative path to its diff; one ANONYMIZED autopsy per gate-FAILED worker (`"a failed candidate: <name>: exit N"`, never a worker id/model name); then the literal instruction to write verdict.json. Build the failed-autopsy list by iterating GATE-FAILED workers — a common bug is iterating the passing/candidate set (always empty), silently dropping the autopsies.
- [2026-06-11, task-002] Verdict validation after the reviewer exits: verdict.json must exist + parse; `outcome` in {adopt, synthesize, escalate}; `candidate` is a valid existing label iff outcome==adopt, else null; `reasoning` is a str. Any failure is an infra error (LoopError -> cli exit 2). CLI exit codes: 0 valid verdict (ANY outcome), 1 gate-everything-failed (no candidates -> no reviewer -> run.json with `review: null`), 2 infra/config (incl. missing reviewer when candidates exist).

## Gotchas
- [2026-06-11, task-001] Derive the repo root from `git rev-parse --show-toplevel`, not `os.getcwd()`/`Path.cwd()`. The worktree (`../<task-id>-w1`) and clean-tree check must be anchored to the git toplevel so the tool works when invoked from a subdirectory.
- [2026-06-11, task-002] Multiple worktrees: `git worktree add` mutates shared repo state and is NOT concurrency-safe — create ALL worktrees serially (with branch/path pre-checks) BEFORE the parallel fan-out. Only the worker COMMANDS run concurrently (ThreadPoolExecutor / Popen), each with CWD set to its own worktree (that CWD isolation is the blindness guarantee). A worker exiting nonzero is recorded, not fatal; gate it anyway.
- [2026-06-11, task-001] Use timezone-aware UTC timestamps: `datetime.now(timezone.utc)`. Avoid `datetime.utcnow()` (deprecated 3.12+, returns naive) and avoid appending a literal `"Z"` to a naive isoformat.
- [2026-06-11, task-001] Never commit `__pycache__/`/`*.pyc`; add a `.gitignore`. Committed bytecode is a recurring hygiene failure and undercuts the `compileall` gate.

## Style
- [2026-06-11, task-001] CLI output is ASCII only — no `✓`/`✗` or other non-ASCII glyphs (terminal/encoding portability).
- [2026-06-11, task-001] Make the loop body callable in-process and testable: `run(...)` returns the run-log dict and accepts an injectable `cwd`; `cli.main(argv=None)` accepts an argv list. Tests should drive these directly rather than spawning `python3 -m farnsworth`.
- [2026-06-11, task-001] Prefer a small typed-error + thin-CLI split (raise a project error in the library, catch it in `cli.main` and print to stderr) over `sys.exit("msg")` for control flow inside library code.

## Per-module notes
### farnsworth/
- [2026-06-11, task-001] Clean decomposition that worked well: `cli` (argparse + exit codes), `config` (load/validate farnsworth.json, built-in default only when path absent), `gitutil` (subprocess git wrappers), `loop`/`runner` (the loop body returning the run-log dict).
- [2026-06-11, task-001] Pre-check branch-exists and worktree-path-exists before `git worktree add` and fail with a clear message, in addition to surfacing git's own nonzero-exit error.
- [2026-06-11, task-002] M2 extends the M1 split cleanly: `config` grows `workers[]` + optional `reviewer` (validate unique, nonempty, filesystem-safe ids; back-compat legacy `worker` -> `[{id:"w1",...}]`; reviewer optional at parse, required-at-runtime only when candidates exist); `gitutil` gains a `diff <base>..<HEAD>` helper; `loop.run()` orchestrates create->dispatch->gate->anonymize->review->validate->write. Keep a `worker_command` shim (first worker's command) if old config tests depend on it. EXTEND the M1 modules, never rewrite from scratch.

### tests/
- [2026-06-11, task-001] Build the scratch repo in a `TemporaryDirectory`; place the brief/config so worktrees (`../<id>-w1`) land inside the temp dir, and force-remove worktrees in teardown. No worktrees, branches, or `.farnsworth/` artifacts may leak into the repo under test.
- [2026-06-11, task-001] The fake worker is a `python3 -c` one-liner that writes a file and commits inside the worktree; tests MUST NOT invoke the real `claude` binary.
- [2026-06-11, task-001] Strong suites assert the briefing ORDERING (tips before task) and the literal `"\n\nTASK: "` separator, the all-gates-run path (failing first gate, passing second still recorded), and that on-disk run.json equals the returned dict.
- [2026-06-11, task-002] Anonymization/autopsy tests must assert the POSITIVE, not just the negative. Checking only `assertNotIn("w1", briefing)` let a candidate ship a briefing where the anonymized autopsy was silently dropped. Also assert the autopsy line IS present (`"a failed candidate:"`), labels ARE present, and no model names leak — and exercise the gate-FAILED-worker path (a mixed two-pass/one-fail run), since a single-passing-worker test never produces an autopsy to check.
- [2026-06-11, task-002] Multi-worker tests must track and force-remove EVERY worktree (`<task>-<id>` per worker, not just `-w1`) in teardown; a fake reviewer is also a `python3 -c` one-liner that writes verdict.json (never the real `claude`), and it should derive the verdict path from its briefing, not a hardcoded task id.
