## Cross-project seed (curated by the orchestrator, 2026-06-12)

These entries are DOMAIN-GENERAL lessons distilled by reviewers on prior
Farnsworth projects (word-garden-1, word-garden-2) and seeded into this
fresh project before round 1 тАФ the loop's first trial of cross-project
memory. They carry their original provenance. From here on, only the
reviewer writes this file.

- When an action TERMINATES a process (game over, job done, connection
  closed), you MUST update every adjacent piece of the contract тАФ status
  message, summary text, final state тАФ not just the boolean flags. The
  recurring defect shape across projects is "flags right, message wrong":
  `won`/`game_over` set correctly while `status_message` still reads as a
  mid-game message. Applies to source AND tests: end-of-process tests MUST
  assert the terminal message/text, not only the flags.
  [2026-06-12, seed; orig. word-garden-2 task-001]
- Tests MUST assert the POSITIVE outcome, not the absence of the negative.
  A "wrong input costs one unit" test asserts the counter went down by
  exactly 1 AND the side effect happened AND the input was recorded тАФ in
  combined assertions. Negative-only assertions have hidden real bugs in
  two prior projects. Applies to all test code.
  [2026-06-12, seed; orig. loop dogfooding task-002, word-garden-1 task-001]
- An end-to-end "success path" test MUST force a known scenario
  (monkeypatch the factory, inject a fixed state or seeded rng) and assert
  the success artifact actually appeared. Asserting only `exit code == 0`
  or `A or B` proves nothing тАФ prior candidates shipped "win" tests that
  never reached a win. Applies to tests.
  [2026-06-12, seed; orig. word-garden-1 task-002]
- Do NOT wrap `argparse.parse_args` in `try/except SystemExit`. `--help`
  must exit 0 and a usage error must exit 2; swallowing `SystemExit` to
  "exit cleanly" turns usage errors into success. This exact defect has
  shipped in two separate projects. Applies to source.
  [2026-06-12, seed; orig. word-garden-1 task-002, word-garden-2 task-002]
- Functions that select or filter from a pool MUST raise (`ValueError`)
  when the pool is empty or the key is unknown. NEVER fall back silently
  to the unfiltered pool тАФ that hides a broken filter behind plausible
  output. Applies to source; tests MUST cover the raising path.
  [2026-06-12, seed; orig. word-garden-2 task-001]
- All I/O and randomness MUST be injectable: `input_fn=input`,
  `output_fn=print`, `rng=None` (module-level `random` when None тАФ never a
  freshly seeded `Random` as the default, which would be deterministic).
  Logic modules stay terminal-free so the suite runs hermetically. Applies
  to source AND tests (tests use the injection points, not patching of
  builtins). [2026-06-12, seed; orig. word-garden-1/2, both tasks]
- Define user-facing message strings ONCE as named module-level constants
  and import them everywhere they are needed тАФ in other source modules AND
  in tests. Re-typed string literals rot silently when wording changes.
  SHARPENED [2026-06-12, task-002]: this binds the UI layer too. Every
  win/loss screen line, every field label, the input prompt, and the
  EOF/interrupt goodbye MUST be a named constant in the module that owns it
  (ui.py / main.py), and tests MUST import and assert against that constant
  тАФ NOT a re-typed substring. Importing a constant and then asserting a
  hand-typed literal of the same text (`from ui import WIN_HEADER` then
  `assertIn("Bloom!", out)`) is the violation we saw and is forbidden:
  import it, assert it. The engine's `MSG_WIN`/`MSG_LOSS` are single-line
  status strings; SPEC ┬з6.6/6.7 screens are multi-line, so you WILL define
  fresh screen constants тАФ define them, don't inline them.
  [2026-06-12, seed; orig. word-garden-1 task-002, word-garden-2 task-002]
- Build test fixtures by constructing the state object directly with
  pinned values. Do NOT call the random factory and then overwrite fields.
  Applies to tests. [2026-06-12, seed; orig. word-garden-1 task-001]
- Hygiene contract: never write or commit `.code-tips.md` (the reviewer
  owns it); never commit bytecode (`__pycache__/`, `*.pyc`); never modify
  `SPEC.md`, `farnsworth.json`, or `tasks/`. Commit all deliverable work
  to your assigned branch. [2026-06-12, seed; orig. loop dogfooding task-001]

## Project тАФ Invariants

- `is_lost(state)` MUST encode "water at 0 BEFORE the win", i.e.
  `return state.remaining_water <= 0 and not is_won(state)`. The bare
  `remaining_water <= 0` form is WRONG: a fully-revealed word at zero water
  then reports won AND lost simultaneously. Correct guesses never cost water,
  so the only states with water 0 are reachable wins-on-last-water and losses;
  the `not is_won` guard is the only thing distinguishing them. Applies to
  source. [2026-06-12, task-001]
- In `apply_guess`, check WIN before LOSS (`if is_won(...): ... elif
  is_lost(...): ...`). A win takes precedence over a water-0 loss on the same
  move. This precedence MUST also hold in the standalone `is_lost` predicate
  (see above) тАФ do NOT rely on apply_guess's ordering to hide a wrong
  predicate; the UI calls `is_won`/`is_lost` directly. Applies to source.
  [2026-06-12, task-001]
- When the guess ENDS the game, `status_message` MUST become the win/loss
  text, overwriting any "Good guess!"/"No match." set earlier in the same
  call. Applies to source. [2026-06-12, task-001]

## Project тАФ Gotchas

- The whole `WORDS` list is length 5тАУ8, so "normal" filtering returns all ten
  and never empties. Easy (4тАУ6) and hard (7+) each keep 5 words. The empty-pool
  `ValueError` branch is therefore UNREACHABLE with the shipped `WORDS` тАФ but it
  MUST still be present (a future word-list edit can empty a window). Applies to
  source. [2026-06-12, task-001]
- Loss text MUST retain SPEC 6.7's "ran out of water" sense, not just "The word
  was: {word}". The player needs to know WHY they lost; an engine that drops the
  reason forces every UI to re-add it. Win/loss message templates SHOULD
  interpolate the secret word (`{word}`). Applies to source. [2026-06-12, task-001]

## Project тАФ Style

- `apply_guess` may either MUTATE-and-return-the-same-`GameState` or return a
  fresh one; both are acceptable. Whichever you choose, the tests MUST match it:
  a mutate-in-place impl with tests that read the returned value works; an
  immutable impl with tests that re-read the input `state` does NOT. State the
  choice in the docstring. Applies to source AND tests. [2026-06-12, task-001]
- Prefer one `_DIFFICULTY` table as the single source of truth for water +
  length window (e.g. `{"easy": (8, 4, 6), "normal": (6, 5, 8),
  "hard": (4, 7, None)}`, `None` = unbounded). Both `water_for` and
  `select_word` read from it. Applies to source. [2026-06-12, task-001]
- Assert messages by full-string equality against the IMPORTED constant
  (`self.assertEqual(s.status_message, MSG_WIN.format(word="GO"))`), not by
  substring (`assertIn("thriving", ...)`) and not via module-attribute access
  (`game.MSG_WIN`). Full equality + import is the strongest rot guard. Applies
  to tests. [2026-06-12, task-001]

## Project тАФ Testing

- The required engine signatures are a CONTRACT for task-002:
  `select_word(difficulty="normal", rng=None) -> str`,
  `water_for(difficulty="normal") -> int`,
  `new_game(difficulty="normal", rng=None) -> GameState`,
  `validate_guess(input_text) -> tuple[str|None, str]`,
  `apply_guess(state, letter) -> GameState`,
  `display_word(state) -> str`, `is_won(state) -> bool`,
  `is_lost(state) -> bool`. `GameState` fields, in order: `secret_word,
  guessed_letters, remaining_water, weed_count, max_water, status_message="",
  game_over=False, won=False`. Applies to source. [2026-06-12, task-001]
- The win/loss PREDICATES MUST be tested DIRECTLY on terminal states, not only
  through `apply_guess`. A round-1 candidate's `is_lost` shipped a real bug
  because its suite only ever reached loss via the apply_guess flow (where win
  is checked first), never calling `is_lost` on a won water-0 state. Add an
  explicit `is_lost(won, water=0) is False` test and an
  `is_won(water=0) is True` test. Applies to tests. [2026-06-12, task-001]
- Test the empty-pool raise by FORCING it (temporarily shrink `WORDS` so a
  difficulty window is empty, in try/finally), not merely by asserting a
  message constant exists. No round-1 candidate did this; it is the field's
  shared coverage gap. Applies to tests. [2026-06-12, task-001]
- Cover multi-occurrence reveal (one guess reveals EVERY position, e.g. "E" in
  SEEDLING -> both E's shown) and the repeated-WRONG-guess no-double-penalty
  case (guessing a wrong letter twice costs water only once). Applies to tests.
  [2026-06-12, task-001]

## Project тАФ UI / CLI (task-002)

- `main`'s signature is EXACTLY `main(argv=None, input_fn=input,
  output_fn=print) -> int`. Honor it character-for-character. Do NOT add
  parameters (a `new_game_fn=` appeared this round purely to serve the
  candidate's own tests) and do NOT make `input_fn`/`output_fn`
  keyword-only with a `*` separator тАФ the brief lists them as
  positional-or-keyword. If a test needs a deterministic word, inject the
  engine's `rng` or monkeypatch `words.select_word` in try/finally; that
  needs NO signature change. Applies to source AND tests. [2026-06-12, task-002]
- SPEC ┬з18 accessibility, EXACT shapes: water carries `count/max_water`
  (`Water: 4/6 тАж`); weeds carry a STANDALONE count (`Weeds: 2 тАж`), NEVER
  `2/max_water` (weeds have no maximum). Both lines ALWAYS show a text label
  AND the numeric count in addition to any symbols тАФ symbols alone fail.
  Applies to source; tests MUST assert the label, the count, and (in emoji
  mode) the symbol count, positively. [2026-06-12, task-002]
- `render` MUST emit SPEC ┬з9 fields in order: title (with growth glyph),
  masked word, guessed letters (sorted, space-separated), water, weeds,
  then the previous turn's `status_message` (omit the message block entirely
  when empty тАФ no dangling blank line). Add at least one test asserting the
  ORDER (positions of the labels), not just presence. Applies to both.
  [2026-06-12, task-002]
- Growth stage MUST be driven by PROGRESS (fraction of distinct secret
  letters revealed), yielding five OBSERVABLE stages тАФ start / progress /
  near-win / win / loss тАФ with five DISTINCT emoji glyphs (ЁЯМ▒ ЁЯМ┐ ЁЯМ╖ ЁЯМ╗ ЁЯеА).
  win/loss come from `state.won` / game-over-and-not-won; the in-between
  three from the revealed-fraction. Do not collapse to only start/win/loss.
  Tests MUST construct a pinned state per stage and assert its glyph.
  Applies to source AND tests. [2026-06-12, task-002]
- `--ascii` MUST make the ENTIRE output ASCII тАФ render AND both end screens.
  Assert it POSITIVELY with `out.isascii()` over a whole scripted game
  including the win and loss screens (not just one render). Conversely,
  assert emoji mode is NOT all-ascii. Use the SPEC ┬з10 fallback (plant `*`,
  water `[****]`, weeds `[---]`); an empty `[]` for zero weeds is acceptable
  but cosmetic. Applies to source AND tests. [2026-06-12, task-002]
- The final-screen branch MUST key on `state.won` (`win_screen if state.won
  else loss_screen`), NOT on re-deriving `is_won`/`is_lost` from the board.
  Re-deriving is currently safe by engine invariant but adds a path where a
  future engine change could print no final screen at all. Applies to source.
  [2026-06-12, task-002]
- Dispatch invalid input through `validate_guess` BEFORE `apply_guess` and
  `continue` without consuming a turn; let the engine's `apply_guess` own
  the repeated-guess no-op. e2e tests MUST prove BOTH no-cost paths: feed
  invalid ("", "ab", "7", "!") then valid letters and assert water is
  unchanged across the invalid ones (e.g. start water still `6/6`); feed the
  same wrong letter twice and assert water dropped exactly once. Applies to
  source AND tests. [2026-06-12, task-002]
- EOF (`EOFError`) and `KeyboardInterrupt` MUST both exit with a friendly
  NAMED-constant goodbye and return 0 тАФ no traceback. Tests MUST inject an
  `input_fn` that raises each and assert the goodbye constant appeared and
  the code is 0. Wrap the loop, not just `parse_args`. Applies to both.
  [2026-06-12, task-002]
- ui.py MUST be pure: every function returns a string; zero `print`/`input`.
  main.py is the ONLY module that touches stdin/stdout, and only via the
  injected `input_fn`/`output_fn`. Applies to source. [2026-06-12, task-002]
- e2e win/loss tests MUST force a KNOWN word (pin via monkeypatched
  `select_word` or injected `rng`) and assert the actual win/loss screen
  text AND the revealed word appeared. Do NOT accept "any pool word" via an
  `any(word in out for word in WORDS)` membership check тАФ that is an
  under-pinned scenario; pin the word and assert it exactly. Applies to
  tests. [2026-06-12, task-002]
