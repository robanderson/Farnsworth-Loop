# Code Tips — Word Garden

Durable, rent-paying lessons for future workers on this project. Every entry carries provenance `[date, task]`. Keep entries as project truths, not incident logs.

## Invariants

- `GameState` fields are a fixed contract from SPEC §12 — exact names AND order: `secret_word, guessed_letters, remaining_water, weed_count, max_water, status_message="", game_over=False, won=False`. `guessed_letters` is `set[str]`. Do not add, rename, reorder, or re-default fields. [2026-06-11, task-001]
- Loss is defined as water reaching 0 *before* the word is revealed. `is_lost` MUST be `remaining_water <= 0 and not is_won(state)` — the `not is_won` guard is mandatory, not optional. A win achieved at exactly 0 water is a WIN, not a loss. [2026-06-11, task-001]
- In `apply_guess`, evaluate WIN before LOSS. A correct guess never changes water/weeds; a wrong guess never reveals the final letter. Checking win first keeps the win-on-last-water case correct. [2026-06-11, task-001]
- `validate_guess` is stateless: it never touches `GameState`. Repeated-guess detection lives in `apply_guess`, not in validation. [2026-06-11, task-001]
- The difficulty table is the single source of truth: easy = water 8 / len 4–6, normal = water 6 / len 5–8, hard = water 4 / len 7+ (no upper bound). Both `water_for` and `select_word` must read from one shared table — never duplicate the length windows in two places. [2026-06-11, task-001]
- `word_garden/` is pure logic: stdlib only, and zero `input()`/`print()`. UI (prompts, rendering, main loop) is task-002 and must not leak into game.py/words.py. [2026-06-11, task-001]

## Gotchas

- `apply_guess` return semantics are NOT pinned by the contract (`-> GameState`). Some implementations mutate in place and return the same object; others return a fresh object. Do not assume the caller's prior reference is preserved after `apply_guess` — always use the returned value, and never assume the input is left untouched. [2026-06-11, task-001]
- A passing mechanical gate (tests + compileall) does not prove correctness of public helpers. The known `is_lost` bug (missing `not is_won`) passed the gate because `apply_guess` checks `is_won` first, hiding it from flag-level tests. Test public helpers directly, including the won-at-zero-water edge. [2026-06-11, task-001]
- `validate_guess` ordering matters: check empty, then length>1, then `isalpha()`. This makes "ab" report a length problem and "7"/"!" report a not-a-letter problem. Running `isalpha()` first would misclassify multi-char alphabetic input. [2026-06-11, task-001]
- Words in `WORDS` are already uppercase, but still `.upper()` defensively when selecting/seeding `secret_word`, since SPEC §6.1 mandates an uppercase internal word and the list may be customised later. [2026-06-11, task-001]

## Style

- Prefer defining user-facing status/validation messages as named module-level constants in `game.py` and reusing them (instead of inlining string literals). This lets tests assert exact strings and lets task-002's UI reuse the same text. [2026-06-11, task-001]
- Keep the difficulty/length config as one dict with `min_len`/`max_len` (use `None` for "no upper bound"); avoid `float('inf')` in an integer length context. [2026-06-11, task-001]
- Build test fixtures by constructing `GameState(...)` directly with a pinned `secret_word`. Do NOT call `new_game()` (random) and then overwrite `.secret_word`/`.guessed_letters` — it is fragile and obscures intent. [2026-06-11, task-001]

## Testing

- Assert the POSITIVE outcome, not just absence of the negative: a wrong-guess test must assert water == before-1 AND weeds == before+1 AND the letter was recorded — ideally in one combined assertion. [2026-06-11, task-001]
- Use an injected `random.Random(seed)` for any test that selects a word; never rely on global RNG for determinism. `select_word`/`new_game` both accept an `rng` parameter for exactly this. [2026-06-11, task-001]
- Cover all ten SPEC §16 cases. For pure logic that means: case 9 = `display_word` masking, case 10 = `select_word` for every difficulty (assert the returned word respects that difficulty's length window). [2026-06-11, task-001]
- Include the multi-occurrence reveal case (e.g. guessing "E" in SEEDLING reveals both E positions) — single-occurrence words won't catch a positional masking bug. [2026-06-11, task-001]

## UI layer (ui.py / main.py)

- Strict split: `ui.py` is PURE — every function takes a `GameState` and returns a `str`, zero `print()`/`input()`. `main.py` is the ONLY module that touches stdin/stdout, and it must take injectable `input_fn=input`/`output_fn=print` and use them exclusively (so e2e tests drive a full game with scripted I/O). `render`/`win_screen`/`loss_screen` take `ascii_mode=False`. [2026-06-11, task-002]
- ASCII mode must be PURE ASCII everywhere — render at every stage AND both end screens. Assert it with `text.isascii()` (or per-char `ord<128`), not just `assertNotIn` of a few emoji. Add a POSITIVE counter-assertion that emoji mode is NOT all-ASCII, so a mode that silently never emits glyphs is caught. [2026-06-11, task-002]
- §18 accessibility = symbol PLUS text, never symbol alone. Water/weeds carry a text label and a count: `Water: 4/6 💧💧💧💧`, `Weeds: 2 🌿🌿`. Weeds is a BARE count (matching the §18 example) — do NOT render weeds as `N/max_water`; weeds are not a fraction of the water budget. The §10 growth stage is itself information: in ASCII mode either give each stage a DISTINCT glyph or (better) accompany the stage with a text label, so the stage is not lost when emoji collapse. A single `*` for all five stages fails this. [2026-06-11, task-002]
- §9 display order is fixed: title, plant/garden status, masked word (`display_word`), guessed letters (sorted, space-joined), water, weeds, then the previous turn's `status_message`. The `Guess a letter: ` prompt belongs to `main.py`, not `render`. Avoid trailing spaces on label lines (e.g. `Weeds:  0 `) — `.rstrip()` the assembled line. [2026-06-11, task-002]
- §10 growth stage is driven by PROGRESS, not turn count: win → loss → near-win → progress → start, checked in that priority. Compute progress from UNIQUE letters revealed (`set(secret) & guessed` over `len(set(secret))`), and decide near-win by letters-missing (e.g. `missing <= 2`) or a ratio threshold — either is fine; pick one and test the boundaries. Reuse the engine's `is_won`/`is_lost`/`game_over` rather than re-deriving win/loss. [2026-06-11, task-002]
- CLI contract: `argparse` with `--difficulty {easy,normal,hard}` (default normal) and `--ascii` (store_true). Do NOT wrap `parse_args` in `try/except SystemExit` — let argparse own exit codes (`--help` → 0, usage error → 2). Swallowing SystemExit to `return 1` breaks both. Handle `EOFError` AND `KeyboardInterrupt` around the input/loop with a friendly goodbye and `return 0` — never a traceback. Invalid input shows the validation message and re-prompts WITHOUT consuming a turn (inner `continue`); test that water is unchanged after invalid input. [2026-06-11, task-002]
- `__main__.py` is tiny: `from .main import main` then `sys.exit(main())`. Running under `python3 -m word_garden` sets `__name__ == "__main__"`, so an `if __name__ == "__main__": sys.exit(main())` guard works too — both forms are fine. [2026-06-11, task-002]
- e2e test pattern: the secret word is random, so either inject a seeded `rng` into `new_game` or monkeypatch `module.new_game` to return a fixed `GameState(...)` (restore in `finally`). Then drive a scripted win and a scripted loss via `input_fn`, capture via `output_fn`, and POSITIVELY assert the final screen text appeared (`Bloom!` / `ran out of water` + the revealed word) and `rc == 0`. [2026-06-11, task-002]

## Per-module notes

- `words.py`: `WORDS` is the SPEC §8 ten-word list, uppercase, order preserved (kept easy to extend). `select_word(difficulty="normal", rng=None)` and `water_for(difficulty="normal")` both raise `ValueError` on unknown difficulty — validate the difficulty key before filtering. [2026-06-11, task-001]
- `game.py`: depends on `words.py` (`select_word`, `water_for`); must stay UI-free. `display_word` returns space-separated tokens, guessed letters shown and others as `_` (e.g. `G _ R _ E _`). `MSG_*` status/validation constants live here — the UI and tests should reuse them, not re-type the strings. [2026-06-11, task-001]
- `tests/` is a package (`tests/__init__.py` present); `python3 -m unittest discover -s tests` must pass from the repo root. New UI tests go in `tests/test_ui.py`. [2026-06-11, task-001]
- Never create/commit `.code-tips.md` from a worker branch (the reviewer owns it), and never modify `SPEC.md`, `farnsworth.json`, `tasks/`, or the engine modules (`game.py`/`words.py`/`__init__.py`). `.gitignore` already excludes `__pycache__/` and `*.pyc` — keep bytecode out of commits. [2026-06-11, task-001]
- Future extensions (SPEC §17 — categories, hints, scores, save/load, colour, daily mode) build on this same split: keep new state in the engine contract, new rendering in `ui.py` (pure), and all new I/O/flags in `main.py`. [2026-06-11, task-002]
