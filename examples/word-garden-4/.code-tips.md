## Testing (binds tests)

- Tests MUST assert the POSITIVE, full observable outcome of a state
  transition — every field that changed, including user-facing message/status
  fields — never just boolean flags or the absence of the negative. Flag-only
  tests have passed real bugs through the mechanical gate twice.
  [seed 2026-06-12; word-garden-1 task-001, word-garden-2 task-001;
  reconfirmed 2026-06-12 task-001 — two gate-passing candidates asserted only
  won/game_over and so missed leaving the wrong terminal status_message]
- Any test that touches randomness MUST inject a seeded `random.Random`;
  never rely on global RNG state for determinism.
  [seed 2026-06-12; word-garden-1 task-001]
- Build test fixtures by constructing the state object directly with pinned
  values. Do NOT call a randomized constructor (e.g. `new_game()`) and then
  overwrite fields — that also leaks global-RNG nondeterminism into the test.
  Equally: NEVER read a value (e.g. the secret word) out of a seeded
  constructor and then drive the system under test through a SEPARATE,
  unseeded entry point — the assertion ends up decoupled from the scenario
  actually running. Pin the randomness in the path under test (patch the
  selector / inject the rng there), not in a throwaway sibling object.
  [seed 2026-06-12; word-garden-1 task-001; reconfirmed 2026-06-12 task-001;
  reconfirmed 2026-06-12 task-002 — a candidate read a Random(42) word from
  `new_game` then ran `main([])` on the global RNG, so its asserted
  "Good guess!" depended on luck, not the pinned word]
- An end-to-end success-path test MUST force a known scenario (inject or
  monkeypatch the source of randomness) and positively assert that the
  success output appeared. Asserting only exit code 0, or an OR of several
  acceptable outputs, proves nothing. A `win OR loss` disjunction is the
  canonical vacuous assertion: EVERY terminating game satisfies it, so it
  passes 100% of runs while testing nothing.
  [seed 2026-06-12; word-garden-2 task-002; reconfirmed 2026-06-12 task-002 —
  a gate-passing candidate's only end-to-end "win" test asserted
  `count("Bloom!")>0 or count("ran out of water")>0` on the global RNG and so
  never actually exercised a win]

## CLI contracts (binds source and tests)

- argparse: NEVER wrap `parse_args` in `try/except SystemExit`. `--help` must
  exit 0 and a usage error must exit 2; swallowing SystemExit breaks both.
  [seed 2026-06-12; word-garden-1 task-002, word-garden-2 task-002]
- Interactive input loops MUST catch `EOFError` AND `KeyboardInterrupt`,
  print a friendly goodbye, and return 0 — never let a traceback reach the
  user. [seed 2026-06-12; word-garden-1 task-002]
- `__main__.py` MUST propagate the entry point's return code
  (`raise SystemExit(main())` or `sys.exit(main())`).
  [seed 2026-06-12; word-garden-2 task-002]

## Design (binds source)

- Strict I/O split: rendering and logic functions return strings and never
  call `print()`/`input()`. Exactly ONE module touches stdin/stdout, and it
  takes injectable `input_fn=input`, `output_fn=print` and uses them
  exclusively. The I/O module MUST call `output_fn` with a SINGLE positional
  string and MUST NOT depend on keyword args of the default sink (e.g. print's
  `end=`/`sep=`); pass any prompt to `input_fn(prompt)`, never emit it via
  `output_fn(prompt, end="")`. A sink injected as a plain `output_fn(text)`
  must work. [seed 2026-06-12; word-garden-1 task-002; reconfirmed
  2026-06-12 task-002 — a gate-passing candidate raised TypeError when given a
  single-arg output_fn because it relied on print's end= kwarg]
- A function accepting `rng=None` MUST default to the module-level `random`
  functions; NEVER construct a fresh seeded `random.Random()` internally
  (that makes production deterministic).
  [seed 2026-06-12; word-garden-2 task-001]
- A filter/selection that yields an empty pool MUST raise `ValueError`;
  NEVER silently fall back to the unfiltered set.
  [seed 2026-06-12; word-garden-2 task-001]
- Define user-facing message strings ONCE as module-level constants; every
  other module AND every test imports them instead of re-typing literals.
  [seed 2026-06-12; word-garden-2 task-001 and task-002;
  reconfirmed 2026-06-12 task-001 — inline literals correlated with the weakest
  candidates]
- Encode any configuration table (limits, bands, tiers) exactly once; two
  copies of the same table is a bug waiting to diverge. This includes a
  difficulty/tier set split across functions: keep the water amount and the
  word-length band for a difficulty in ONE table, not separate dict + if/elif.
  [seed 2026-06-12; word-garden-1 task-001; reconfirmed 2026-06-12 task-001]

## Game logic (binds source and tests)

- A state transition that ENDS the game (win or loss) MUST overwrite
  `status_message` with that terminal screen's text (SPEC §6.6 win / §6.7 loss),
  NEVER leave the ordinary correct/wrong-guess message in place. The terminal
  message MUST name the secret word wherever the spec's end screen shows it.
  Tests MUST assert the exact terminal message, not only `won`/`game_over`.
  [2026-06-12, task-001]
- `apply_guess(state, letter) -> GameState` is contracted only to RETURN the
  updated state; implementations may mutate in place or return a fresh object.
  Therefore callers MUST reassign (`state = apply_guess(state, letter)`) and
  MUST NOT assume the input object is left unchanged. (binds source and tests)
  [2026-06-12, task-001]

## UI / rendering (binds source and tests)

- An ASCII / no-emoji fallback MUST preserve every INFORMATION CHANNEL the rich
  mode carries, not just strip the glyphs. If a status indicator (growth stage,
  health, progress) VARIES across states in emoji mode, its ASCII form MUST vary
  too — NEVER collapse all states to a single constant glyph. The fallback
  exists for the users who can't see emoji; flattening it deletes the very
  signal it should carry. Tests MUST assert the ASCII indicator differs between
  at least two distinct states, not only that output is pure ASCII.
  [2026-06-12, task-002 — a gate-passing candidate mapped every growth stage to
  "*" in ASCII mode, so start/progress/near-win were indistinguishable]
- A status row that conveys a resource or count MUST carry a text LABEL and a
  numeric COUNT, optionally plus symbols — NEVER symbols/colour alone
  (SPEC §18). Assert the label AND the count positively, in BOTH render modes.
  [2026-06-12, task-002]
- A multi-line end SCREEN (e.g. the win/loss screen) is a DIFFERENT artifact
  from the engine's one-line `status_message`. The UI layer MAY own its own
  screen strings, but they MUST still be defined ONCE as constants and imported
  by tests (never re-typed), and the loss screen MUST reveal the secret word.
  [2026-06-12, task-002]
- Render the full-turn display in the spec's stated element order and emit each
  enumerated element (title, plant/garden status, masked word, guessed letters,
  water, weeds, previous status). Computing "near-win"/progress MUST use the set
  of DISTINCT secret letters, not raw positions, so words with repeated letters
  are classified correctly. [2026-06-12, task-002]
