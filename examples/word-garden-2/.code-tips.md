# Code Tips — Word Garden (durable lessons for future workers)

Provenance tag format: `[YYYY-MM-DD, task-NNN]`. These are project contracts.

## Invariants

- `is_lost(state)` MUST be `state.remaining_water <= 0 and not is_won(state)`.
  Win strictly beats loss; a fully-revealed word at 0 water is a WIN, never a
  loss. `[2026-06-12, task-001]`
- `is_won(state)` MUST be a pure predicate over `guessed_letters` vs
  `secret_word` only; it MUST NOT read water or weeds. `[2026-06-12, task-001]`
- `new_game` MUST set `remaining_water == max_water == water_for(difficulty)`,
  `weed_count == 0`, `guessed_letters == set()`, `game_over == won == False`, and
  `secret_word` uppercased. `[2026-06-12, task-001]`
- A correct guess MUST NOT change `remaining_water` or `weed_count`. A wrong
  guess MUST change each by exactly 1 (water -1, weeds +1) and record the letter.
  `[2026-06-12, task-001]`

## Gotchas

- When a guess ENDS the game, `apply_guess` MUST overwrite `status_message` with
  the win or loss message — not leave the "Good guess!"/"No match. A weed
  appears." text. Two of five candidates set `won`/`game_over` but forgot the
  message; flag-only tests did not catch it. `[2026-06-12, task-001]`
- `select_word` MUST raise `ValueError` when the length filter yields no words.
  MUST NOT fall back to the full `WORDS` list — that silently returns out-of-band
  words and hides a broken difficulty band. `[2026-06-12, task-001]`
- `select_word`/`new_game` with `rng=None` MUST use the module-level `random`
  functions (`random.choice`). MUST NOT construct a fresh seeded `random.Random`
  (that would be deterministic). `[2026-06-12, task-001]`
- Unknown difficulty MUST raise `ValueError` in BOTH `select_word` and
  `water_for`, checked BEFORE building the candidate pool. `[2026-06-12, task-001]`

## UI layer (ui.py / main.py / __main__.py)

- `ui.py` is PURE: every function returns a string, zero `print`/`input`. Keep a
  test that asserts the ui source contains no `print(`/`input(` (inspect.getsource).
  `main.py` is the ONLY module touching stdin/stdout; it MUST use the injected
  `input_fn`/`output_fn` exclusively (signature `main(argv=None, input_fn=input,
  output_fn=print) -> int`). `[2026-06-12, task-002]`
- `render` order is FIXED (SPEC §9): title -> plant/garden status -> masked word
  (`display_word`) -> guessed (sorted, space-joined) -> water -> weeds -> previous
  `status_message`. Growth-stage layout is flexible (fold the stage glyph into the
  title OR a separate `Plant:` line) but the stage MUST actually change with
  progress — a static title glyph with no Plant line shows no growth.
  `[2026-06-12, task-002]`
- §18 accessibility, NON-NEGOTIABLE: water shows `remaining/max` (e.g. `4/6`) AND a
  text label AND symbols. Weeds shows a STANDALONE count (e.g. `Weeds: 2 ...`) AND
  label AND symbols — weeds is its OWN integer, NEVER a fraction of water
  (`2/6` for weeds is WRONG). `[2026-06-12, task-002]`
- ASCII mode MUST be 100% ASCII everywhere — render at every stage AND the win/loss
  end screens. Prefer writing emoji as `\\U0001F33B` escapes so the source file
  itself stays ASCII-safe and no stray literal glyph can leak. `[2026-06-12, task-002]`
- Expose growth-stage as a PUBLIC `growth_stage(state) -> str` and unit-test it
  directly (start/progress/near-win/win/loss). Compute win/loss from the engine
  `is_won`/`is_lost` predicates, NOT by re-deriving from water — this preserves the
  win-beats-loss-at-zero-water invariant. `[2026-06-12, task-002]`
- `loss_screen` MUST reveal `secret_word`; `win_screen` MUST show it too (SPEC
  §6.6/6.7). `[2026-06-12, task-002]`

## CLI / argparse / loop

- Use `argparse` with `--difficulty {easy,normal,hard}` (default normal, via
  `choices=`) and `--ascii` (store_true). Do NOT wrap `parse_args` in
  `try/except SystemExit`: `--help` must exit 0 and a usage error (bad value /
  unknown flag) must exit 2. Swallowing `SystemExit` to "exit cleanly" turns a
  usage error into exit 0 and is a bug (one of three candidates did this).
  `[2026-06-12, task-002]`
- Invalid input: show the validation message and `continue` WITHOUT calling
  `apply_guess`, so no water/weed/turn is consumed. `[2026-06-12, task-002]`
- Catch `EOFError` (Ctrl-D) AND `KeyboardInterrupt` (Ctrl-C) — a single
  `except (EOFError, KeyboardInterrupt)` around the loop is cleanest — print a
  friendly goodbye and `return 0`. Never let a traceback reach the user.
  `[2026-06-12, task-002]`
- `__main__.py` does `raise SystemExit(main())` (or `sys.exit(main())`) so
  `python3 -m word_garden` returns main's exit code. `[2026-06-12, task-002]`

## Style

- `apply_guess(state, letter)` MUST treat its return value as authoritative.
  Standardize on MUTATE-IN-PLACE-AND-RETURN-`state` (the adopted contract).
  Callers MUST use the returned object and MUST NOT assume the input is left
  untouched. If you instead return a copy, copy `guessed_letters` too — never
  share the set by reference in only some branches. `[2026-06-12, task-001]`
- Define the five user-facing messages as module-level constants in `game.py`
  (`MSG_CORRECT`, `MSG_WRONG`, `MSG_WIN`, `MSG_LOSS`, `MSG_GAME_OVER`) so the UI
  imports them instead of string-matching. `[2026-06-12, task-001]`
- SHARPENED: this means UI TESTS too. When a test asserts an engine feedback
  message appeared, import and assert `game.MSG_CORRECT` etc. — do NOT paste the
  literal `"Good guess! The garden grows."`. Likewise expose any UI-owned constant
  the test needs (e.g. the goodbye line as `GOODBYE` in main.py) and import it
  rather than re-typing. Two of three task-002 candidates re-typed the strings;
  hardcoded literals silently rot when the engine wording changes.
  `[2026-06-12, task-002]`
- `WORDS` MUST be the exact 10 uppercase words from SPEC section 8, in order.
  `[2026-06-12, task-001]`
- `word_garden/__init__.py` SHOULD re-export the public API (`GameState`,
  `new_game`, `validate_guess`, `apply_guess`, `display_word`, `is_won`,
  `is_lost`, `WORDS`, `select_word`, `water_for`). `[2026-06-12, task-001]`

## Testing

- End-of-game tests MUST assert the terminal `status_message`, not only `won`
  and `game_over`. `[2026-06-12, task-001]`
- Include a "win on the last unit of water" test: complete the word while
  `remaining_water == 0` and assert `won` True and `is_lost` False.
  `[2026-06-12, task-001]`
- Multi-occurrence reveal MUST be tested with a repeated-letter word (e.g.
  `LEVEL`, `SEEDLING`): one guess reveals ALL positions. `[2026-06-12, task-001]`
- `select_word` MUST be sampled many times per difficulty with an injected
  `random.Random` and asserted in-`WORDS`, uppercase, and within the length band
  (easy 4-6, normal 5-8, hard 7+). `[2026-06-12, task-001]`
- Tests live in `tests/` as a package; `python3 -m unittest discover -s tests`
  MUST be green from repo root. `[2026-06-12, task-001]`
- A scripted end-to-end `main()` "win" test MUST actually FORCE a known word
  (monkeypatch `new_game`/`select_word` or inject a fixed `GameState`) and assert
  the win screen + word appeared. Asserting only `code == 0` or
  `"Bloom!" OR "Goodbye"` proves nothing — two task-002 candidates shipped a win
  test that never reached a win. `[2026-06-12, task-002]`
- UI tests MUST include POSITIVE assertions both ways: ascii output
  `assert rendered.isascii()` over a WHOLE game incl. end screens, AND emoji output
  `assert not rendered.isascii()` (proves the modes differ). Assert the
  invalid-input path leaves `remaining_water` unchanged, not just that a message
  printed. `[2026-06-12, task-002]`

## Per-module notes

- `words.py`: stdlib only. Encode the difficulty table once
  (water, min_len, max_len); use `None`/`float("inf")` for the hard upper bound.
  `[2026-06-12, task-001]`
- `game.py`: `validate_guess(input_text)` MUST NOT consult game state (no
  repeated-guess check here — that belongs to `apply_guess`). Order checks
  empty -> length!=1 -> not alpha, returning a specific message each. Guarding
  `None` input is acceptable but not required (the UI feeds strings).
  `[2026-06-12, task-001]`
- `game.py`: `display_word` returns guessed letters else `_`, space-joined over
  `secret_word` (e.g. `G _ R _ E _`). `[2026-06-12, task-001]`
- `GameState` field names/order MUST match SPEC section 12 exactly:
  `secret_word, guessed_letters, remaining_water, weed_count, max_water,
  status_message="", game_over=False, won=False`. `[2026-06-12, task-001]`
